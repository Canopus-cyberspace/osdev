No kernel code was changed for this analysis.

## Top Level Flow

The official wrapper performs four major stages:

1. `prework.py`
   - Runs `make all` in the submitted repository.
   - Sets the SBI firmware field to `default`.
   - Creates one testcase worth 100 wrapper points.
2. `run.py`
   - Copies compressed sdcard images from `testdata` into the submit directory.
   - Starts RISC-V and LoongArch QEMU runners in parallel.
   - Parses each architecture serial log with the judge scripts.
3. `judge/*.py`
   - Each judge receives only the serial lines between official test group markers.
   - The judge returns a per-suite score table.
4. `postwork.py`
   - Sums every per-suite score from both architectures.
   - Emits a rank table with keys such as `basic-musl-rv` and `basic-musl-la`.
   - Reports verdict string `Accpted` even when the numeric score is `0`.

`Accpted` therefore means the wrapper completed. It does not mean any test suite passed.

## QEMU Commands

The RISC-V command generated by `run_qemu.py` is:

```text
qemu-system-riscv64 -machine virt -kernel kernel-rv -m {mem} -nographic -smp {smp} -bios default -drive file=sdcard-rv.img,if=none,format=raw,id=x0 -device virtio-blk-device,drive=x0,bus=virtio-mmio-bus.0 -no-reboot -device virtio-net-device,netdev=net -netdev user,id=net -rtc base=utc
```

With the current `testdata/config.json`, this resolves to:

```text
qemu-system-riscv64 -machine virt -kernel kernel-rv -m 1G -nographic -smp 1 -bios default -drive file=sdcard-rv.img,if=none,format=raw,id=x0 -device virtio-blk-device,drive=x0,bus=virtio-mmio-bus.0 -no-reboot -device virtio-net-device,netdev=net -netdev user,id=net -rtc base=utc
```

If `disk.img` exists, RISC-V also receives:

```text
-drive file=disk-rv.img,if=none,format=raw,id=x1 -device virtio-blk-device,drive=x1,bus=virtio-mmio-bus.1
```

The LoongArch command generated by `run_qemu.py` is:

```text
qemu-system-loongarch64 -kernel kernel-la -m {mem} -nographic -smp {smp} -drive file=sdcard-la.img,if=none,format=raw,id=x0 -device virtio-blk-pci,drive=x0 -no-reboot -device virtio-net-pci,netdev=net0 -netdev user,id=net0 -rtc base=utc
```

With the current `testdata/config.json`, this resolves to:

```text
qemu-system-loongarch64 -kernel kernel-la -m 1G -nographic -smp 1 -drive file=sdcard-la.img,if=none,format=raw,id=x0 -device virtio-blk-pci,drive=x0 -no-reboot -device virtio-net-pci,netdev=net0 -netdev user,id=net0 -rtc base=utc
```

If `disk.img` exists, LoongArch also receives:

```text
-drive file=disk-la.img,if=none,format=raw,id=x1 -device virtio-blk-pci,drive=x1
```

## Sdcard Image Preparation

The official harness keeps compressed images in `/home/lenovo/oscomp-official-env/testdata`:

```text
sdcard-rv.img.gz
sdcard-la.img.gz
sdcard-rv.img.xz
sdcard-la.img.xz
```

For the actual run, `run.py` copies the `.gz` image into the submit directory. The RISC-V runner then decompresses with `gzip -d sdcard-rv.img.gz`; the LoongArch runner uses `gzip -df sdcard-la.img.gz`.

The images are raw ext4 filesystems, not partitioned disk images. The official harness does not mount them for the kernel. It only exposes them to QEMU as virtio block devices. The kernel must discover the virtio block device, read the raw ext4 filesystem, and execute programs from it.

The decompressed images are 4096 MiB. The RISC-V image root contains:

```text
/glibc
/musl
/lost+found
```

The user programs and suite scripts are inside `/glibc` and `/musl`.

## Official Serial Framing

The current parser in `run.py` looks for exact official group markers:

```text
#### OS COMP TEST GROUP START <group> ####
...
#### OS COMP TEST GROUP END <group> ####
```

The start regex is:

```text
#### OS COMP TEST GROUP START ([a-zA-Z0-9-]+) ####
```

Only lines between a matching start marker and an end marker are piped into the corresponding judge script. If the kernel never prints a group marker, the harness still invokes every judge with empty input, and every suite scores zero.

The suite scripts inside the image already print these markers. For example, `/musl/basic_testcode.sh` prints:

```text
#### OS COMP TEST GROUP START basic-musl ####
```

then runs `./basic/run-all.sh`, then prints:

```text
#### OS COMP TEST GROUP END basic-musl ####
```

## Judge Output Expectations

The basic judges parse sections with this shape:

```text
========== START <test_name> ==========
...
========== END <test_name> ==========
```

Each basic test has strict expected body text. Examples:

- `write`: expects `Hello operating system contest.`
- `read` and `open`: expect `Hi, this is a text file.` and `syscalls testing success!`
- `getpid`: expects `getpid success.` and a positive `pid = <n>`.
- `execve`: expects `I am test_echo.` and `execve success.`
- `waitpid`: expects `This is child process`, `waitpid successfully.`, and `wstatus: 3`.
- `yield`: expects 15 body lines with output from tasks `0`, `1`, and `2`.

The BusyBox judges parse:

```text
testcase <name> success
testcase <name> fail
```

The Lua judges parse:

```text
testcase lua <script> success
testcase lua <script> fail
```

The libctest judges parse:

```text
START entry-static.exe <case>
Pass!
START entry-dynamic.exe <case>
Pass!
```

The LTP judges parse LTP-specific case lines and summary/pass markers. The performance judges parse benchmark-specific result sections and compare measured values against baselines.

## LoongArch Status

The official environment always launches both RISC-V and LoongArch and includes both in the rank table. However, the score is a sum of independent per-architecture group scores. A nonzero RISC-V score is possible even if LoongArch remains zero.

The current compatibility copy named `kernel-la` is not a real LoongArch kernel. The latest official log reports:

```text
qemu-system-loongarch64: could not load kernel 'kernel-la': Failed to load ELF
```

That means LoongArch contributes zero in this environment until a real LoongArch kernel is produced. It should not be represented as implemented.
